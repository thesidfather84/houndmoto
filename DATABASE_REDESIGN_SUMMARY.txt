HOUNDMOTO DATABASE REDESIGN - EXECUTIVE SUMMARY
================================================================================

PROBLEM:
--------
Your data is scattered across 7+ JS files with NO centralized vehicle record.
This works for 50 vehicles, but will be a nightmare at 300+ vehicles.

CURRENT STATE:
- vehicleDirectory.js (851 vehicles, recognition only)
- vehicleCoverageData.js (specs as strings, ~40 vehicles complete)
- fluidDatabase.js (DUPLICATE specs)
- vehicleDtcKnowledge.js (vehicle-specific DTC for only 20 vehicles)
- dtcCodes.js (generic DTC codes, ~400)
- manualRefsData.js (manual links)
- partsData.js (generic parts, not vehicle-specific)
- diagnosisWizardData.js (symptom trees, not vehicle-specific)
- tipsData.js (user tips)

PROBLEMS WITH CURRENT STRUCTURE:
1. No single vehicle ID → loose string matching
2. Specs duplicated in multiple files
3. No structured failures → comma-separated strings
4. No torque specs anywhere
5. No repair procedures anywhere
6. No parts cross-references
7. Vehicle-specific DTC only for 20 vehicles
8. Hard to add new data types
9. No data quality tracking
10. Will require major rework when you add hundreds of vehicles

SOLUTION:
---------
Redesign into unified Postgres database with 10 normalized tables:

1. vehicles (851 rows) - Core vehicle record
2. vehicle_generations (3000 rows) - Generation per vehicle
3. vehicle_specs (3000 rows) - All specs: oil, fluid, tire, battery, bulbs, etc.
4. vehicle_common_failures (5000 rows) - What breaks, costs, symptoms
5. vehicle_dtc_mappings (15000 rows) - Vehicle-specific DTC codes
6. vehicle_maintenance (6000 rows) - Maintenance schedule
7. vehicle_torque_specs (2000 rows) - Torque for every component
8. vehicle_repair_procedures (500 rows) - Step-by-step procedures
9. vehicle_parts (2000 rows) - Part compatibility, cost, lifespan
10. vehicle_manuals (500 rows) - External manual references

ADVANTAGES:
-----------
✅ Single source of truth (no duplicates)
✅ Structured data (arrays instead of strings)
✅ Scalable (indexed queries, even at 10,000+ vehicles)
✅ Room for everything (torque, procedures, parts, images)
✅ Data quality tracking (verified flags, confidence levels)
✅ Admin analytics (coverage analyzer, gap detection)
✅ Fast queries (< 1 second even with 37,000 rows)
✅ Easy to extend (just add columns or tables)

QUESTIONS YOU MIGHT HAVE:

Q: Can I keep the current structure?
A: You can, but at 100+ vehicles you'll regret it. Torque specs and procedures
   are impossible. Analytics are hard. Every query requires multi-file lookups.

Q: How long will migration take?
A: Most of your data (50 vehicles) will migrate automatically. Testing and
   validation take the time, not the migration itself.

Q: Will this break existing functionality?
A: No. You'll create new tables, migrate in parallel, test thoroughly, then
   flip the switch. Old files can stay until you're confident.

Q: What if I discover issues mid-migration?
A: You won't. The migration is deterministic - same vehicle, same data, just
   restructured. Test with 5 vehicles first.

Q: Is this worth the 2-3 weeks?
A: Yes. You're about to add 50-100 vehicles. Doing it right now saves 10x time
   later and enables features (torque, procedures) that are impossible now.
